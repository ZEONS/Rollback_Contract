# [프로젝트 기획서] 단일 통합 화면(Single View) 기반 계약·청구·수납 데이터 프로세스 동기화 채널 구축

---

## 1. 프로젝트 배경 및 목적

### 1.1 배경

* **결산 마감으로 인한 데이터 잠금(Lock):** 이지램스(ezREMS) 시스템은 결산 정합성을 위해 계약 만료 시 데이터가 마감 처리되어 임의 수정이 불가능합니다.
* **하위 데이터 종속성 복잡도:** 계약 만료 이후 발생한 실비 관리비, 청구/수납, 세금계산서, 보증금 정산 등 다양한 하위 트랜잭션 데이터로 인해 데이터 복원 및 수정 시 누락 오류 리스크가 상존합니다.
* **이해관계자 간 소통 파편화:** 사용자, 운영자, 개발자가 동일한 데이터 상태를 보지 못해 비대면 프로세스 조율 시 많은 의사소통 비용(커뮤니케이션 미스)이 발생하고 있습니다.

### 1.2 목적

> **사용자, 운영자, 개발자가 '단일 통합 화면(Single View)'을 통해 계약 마감 취소 및 데이터 변경 흐름을 실시간으로 투명하게 공유하고, 정합성이 검증된 데이터 수정 프로세스를 안전하게 이행할 수 있는 소통 기반을 구축합니다.**

---

## 2. 핵심 요구사항 및 기능 명세

### 2.1 원스크린 데이터 종속성 시각화 (One-Screen Dependency Viewer)

* 계약 만료 상태를 해제하기 전, 그 이후 파생된 6대 추가 데이터(실비, 청구, 수납, 보증금 정산, 세금계산서, 현금영수증)의 존재 여부와 상태를 하나의 화면에서 트리 구조로 직관적으로 시각화합니다.

### 2.2 하위 데이터 단계별 롤백(Rollback) 가이드 가동

* 데이터 간의 정합성이 깨지지 않도록 하위 데이터(영수증/계산서)부터 상위 데이터(계약 마스터) 순으로 역순 정정 및 삭제 처리가 가능한 프로세스 가이드를 제공합니다.
* **데이터 삭제/취소 권한 제어:** 운영자 및 관리자가 안전하게 선행 데이터를 확인한 후 삭제 또는 발행 취소를 수행할 수 있는 인터페이스를 포함합니다.

## 4. 핵심 프로세스 흐름 (Workflow)

```
[1단계: 수정 요청 및 상태 감지]
사용자가 '한 화면'에서 계약 만료 취소/수정 요청 ➔ 시스템이 계약 번호 기준 하위 데이터 스캔

[2단계: 종속성 데이터 사전 체크]
추가 데이터(실비/청구/수납/증빙 등) 존재 여부 실시간 Alert 
 ➔ 추가 데이터가 없는 경우: 즉시 계약만료 플래그 해제 후 수정 가능 상태로 전환
 ➔ 추가 데이터가 있는 경우: 3단계 이행

[3단계: 관리자 승인 및 역순 취소(Rollback)]
관리자가 단일 화면에서 하위 데이터(세무 증빙 ➔ 수납 ➔ 청구 ➔ 정산) 순으로 안전하게 취소/삭제 처리

[4단계: 마스터 복원 및 이력 동기화]
상위 테이블(re_cont_master, re_room_master)의 이력 컬럼(date_chg)을 최종 업데이트하여 데이터 무결성 완료

```

## 5. UI/UX 디자인 가이드라인

* **정보 격차 해소 디자인:** 비전문가인 '사용자'도 결산 마감의 개념을 이해할 수 있도록 "이 계약은 뒤에 발행된 청구서가 있어 즉시 수정할 수 없습니다"와 같은 직관적인 안내 UI를 제공합니다.
* **신호등 지표(Signaling):** 역순 삭제가 필요한 대상을 빨간색(삭제 필요), 노란색(확인 필요), 초록색(정상 취소 완료) 상태 코드로 단일 통합 화면에 매핑하여 개발자와 운영자 간의 불필요한 질의응답 회수를 0회로 수렴시킵니다.

---

## 6. 기대 효과

* **의사소통 비용의 획기적 절감:** 사용자, 운영자, 개발자가 동일한 화면에서 실시간 피드백을 공유하므로 "왜 수정이 안 되나요?"라는 반복적 질의에 대응하는 리소스가 소멸합니다.
* **데이터 무결성(Data Integrity) 보장:** 결산 마감 로직을 안전하게 우회하여 하위 트랜잭션 데이터를 순차적으로 초기화하므로, 데이터 정합성이 깨져 발생하는 야간 긴급 핫픽스(Hotfix) 공수를 원천 차단합니다.
* **운영 투명성 제고:** 데이터가 어떤 사유로 취소되고 변경되었는지 `date_last_chg` 이력 내역이 단일 소통 화면에 투명하게 누적되어 향후 자산 운영 감사(Audit) 추적이 용이해집니다.
