프로젝트 요약

프로젝트명계약만료 취소 롤백 시스템 (Rollback Simulator)
도메인부동산 임대차 관리 플랫폼 — 계약 관리 모듈
문제 정의결산 마감(lock)된 계약 만료 상태를 직접 수정할 수 없어 담당자가 수동으로 대응 중이며, 순서 오류 시 데이터 정합성이 깨짐
솔루션역순 롤백 가이드 및 단계별 실행 UI를 제공하여, 처리 순서를 강제하고 처리 상태를 실시간으로 시각화
대상 사용자관리자(ADMIN), 슈퍼관리자(SUPER_ADMIN)
주요 제약결산 마감 테이블 직접 수정 불가 / 국세청 API 취소 선행 필수 / 역순 처리 순서 위반 불가
이 문서는 기획서(Functional Spec)와 기술 스펙(Technical Spec)을 통합한 단일 레퍼런스입니다. 개발·QA·운영 팀이 공통 참조하도록 설계되었습니다.

배경 및 목적

왜 이 시스템이 필요한가

현황 문제

현재 계약만료 취소는 100% 수동 처리로, 담당자가 SQL을 직접 실행합니다. 처리 순서 실수로 인한 데이터 정합성 오류가 월 평균 2~3건 발생하고 있습니다.
AS-IS (현재)
  • 수동 SQL 실행 — 순서 강제 없음
  • 현금영수증/세금계산서 누락 취소 발생
  • 처리 이력 비체계적 (메모 수준)
  • 부분 취소 시 롤백 불가
  • 처리 후 마스터 상태 미갱신 사례 존재
TO-BE (목표)
  • 6단계 역순 롤백 순서 강제
  • 외부 API 자동 연동 (현금영수증·세금계산서)
  • 단계별 처리 이력 자동 감사 로그
  • 트랜잭션 묶음 — 중간 실패 시 원상복구
  • 마스터 상태 자동 갱신 및 잠금 해제

목표 및 성공 지표

처리 오류율 감소목표: 현재 대비 95% 감소
처리 소요시간 단축목표: 평균 40분 → 5분 이내
감사 로그 커버리지목표: 100%
담당자 교육 없는 자가 처리율목표: 80% 이상

기능 요구사항(FR) · 비기능 요구사항(NFR)

기능 요구사항

FR-01 계약 조회 필수
  • 계약번호(cont_id) 입력으로 re_cont_master 조회
  • 마감 잠금(lock_flag = 'Y') 상태 여부를 UI에 표시
  • 계약 기간, 만료일, 보증금, 최종 변경일 표시
  • 잘못된 계약번호 입력 시 명확한 오류 메시지 반환
FR-02 종속 데이터 체크 필수
  • 5개 연관 테이블의 잔존 데이터 건수 자동 조회
  • 각 항목별 처리 상태 표시: 삭제 필요(빨강) / 확인 필요(노랑) / 완료(초록)
  • 종속 데이터가 없는 경우 즉시 마스터 복원 단계로 이동
FR-03 역순 롤백 처리 핵심
  • 현금영수증 → 세금계산서 → 수납 → 청구 순서 강제
  • 선행 단계 미완료 시 다음 단계 실행 차단
  • 각 단계별 처리 안내 텍스트 및 대상 테이블 명시
  • 처리 완료 확인 버튼으로 상태를 green으로 전환
FR-04 마스터 잠금 해제 필수
  • 모든 종속 데이터 취소 완료 후 해제 버튼 활성화
  • re_cont_master의 lock_flag, date_chg, date_last_chg 갱신
  • 처리 완료 모달에 SQL 실행 결과 및 타임스탬프 표시
FR-05 처리 이력 로그 권장
  • 각 단계 완료 시 타임스탬프·처리자·대상 계약번호 로그 기록
  • tb_rollback_audit_log 테이블에 영구 저장
  • UI 내 실시간 로그 피드 표시
  • 로그 CSV 내보내기 기능

단계별 세부 처리

STEP 1 종속 데이터 전수 조회
계약번호로 5개 테이블 JOIN 조회 → 건수 집계 → UI 트리 렌더링
STEP 2 현금영수증 자진취소
tb_cash_receipt 건별 국세청 자진취소 API 호출 → tb_cash_temp.status_cd = 'CANCEL' UPDATE
STEP 3 세금계산서 수정발행
tb_tax 승인번호 확인 → 마이너스 세금계산서 발행 API → tb_tax_line 연동
STEP 4 수납 및 원장 취소
re_bill_receipt_acc_if DELETE → re_bill_receipt.status = 'CANCELLED' UPDATE (트랜잭션)
STEP 5 청구 데이터 역추적 삭제
re_bill_request_acc_if DELETE → re_bill_request WHERE status='BILLED' DELETE (트랜잭션)
STEP 6 마스터 복원 및 잠금 해제
re_cont_master lock_flag='N', status='ACTIVE', date_chg=NOW() UPDATE → re_room_master 공실 처리

CREATE TABLE tb_rollback_audit_log (
  log_id        NUMBER        NOT NULL,   -- PK (SEQ)

  step_cd       VARCHAR2(20)  NOT NULL,   -- CASH/TAX/RECEIPT/BILL/MASTER

  err_msg       VARCHAR2(1000),
  processed_by  VARCHAR2(50)  NOT NULL,   -- 처리자 ID
  processed_at  TIMESTAMP     DEFAULT SYSTIMESTAMP,
  ip_addr       VARCHAR2(45),               -- 요청 IP
  CONSTRAINT pk_rollback_log PRIMARY KEY (log_id)
);

처리 순서 검증 로직

-- 역순 처리 강제 검증 (서버 사이드)
const ROLLBACK_ORDER = ['CASH', 'TAX', 'RECEIPT', 'BILL'];

function validateOrder(requestedStep, stepStatus) {
  const idx = ROLLBACK_ORDER.indexOf(requestedStep);
  if (idx > 0) {
    const prevStep = ROLLBACK_ORDER[idx - 1];
    if (stepStatus[prevStep] !== 'DONE') {
      throw new RollbackOrderViolationError(prevStep);
    }
  }
}

트랜잭션 그룹 설계

TX-1: 현금영수증

외부 API 완료 후 내부 UPDATE

  • CALL sp_cash_receipt_cancel()
  • UPDATE tb_cash_receipt
  • UPDATE tb_cash_temp
TX-2: 수납·청구 (묶음)

BEGIN / COMMIT 단일 트랜잭션

  • DELETE re_bill_receipt_acc_if
  • UPDATE re_bill_receipt
  • DELETE re_bill_request_acc_if
  • DELETE re_bill_request
TX-3: 마스터 복원

마지막 단계, 잠금 해제

  • UPDATE re_cont_master
  • UPDATE re_room_master
  • INSERT tb_rollback_audit_log
멱등성 보장: 각 단계는 이미 취소된 데이터를 다시 처리해도 오류가 발생하지 않도록 WHERE status != 'CANCELLED' 조건을 포함합니다.

  • 불변 로그: 감사 로그는 INSERT ONLY — UPDATE/DELETE 불가 (DB 트리거로 강제)
  • IP 화이트리스트: 내부망 IP 대역만 허용 (외부 접근 차단)
  • SQL Injection: 모든 파라미터 Prepared Statement 처리 (MyBatis #{} 사용)
